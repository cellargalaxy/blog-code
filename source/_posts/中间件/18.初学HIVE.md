## 行存储与列存储

+ OLTP = On-Line Transaction Processing (联机事务处理)
+ OLAP = On-Line Analytical Processing (联机分析处理)

### 行存储

+ 同一行的数据会被连续存储，在OLTP中有优势
+ 优点
    + 数据写入常见是按行写入，因此数据写入速度快，适合事务型操作
    + 由于同一行的数据会被连续存储，整行数据读取的效率高
+ 缺点
    + 如果只查询部分列，由于需要整行读取，数据读取效率较低
    + 同一行多种类型的数据连续存储，导致压缩率较低
+ 例如：TextFile/SequenceFile

### 列存储

+ 同一列的数据会被连续存储，在OLAP中有优势
+ 优点
    + 查询少量列，同一列的数据存储在一起，读取速度与IO效率效率高
    + 相同类型的数据被连续存储，压缩率高
+ 缺点
    + 数据写入常见是按行写入，数据按列存储，会导致写入较慢IO较多
    + 读取整行的效率较低
+ 例如：ORC/Parquet

## 存储格式

HIVE表的底层数据，作为一个文件，通常存储在HDFS上，常见的文件格式如下

| 类型           | 存储方式 |
|--------------|------|
| TextFile     | 行存储  |
| SequenceFile | 行存储  |
| RCFile       | 行列混合 |
| Parquet      | 列存储  |
| ORC          | 列存储  |

### TextFile

TextFile是HIVE默认的存储格式，数据以文本形式的行存储。
格式类似于CSV，使用分隔符分隔列（分隔符可选），使用换行符分隔行。
TextFile是Hadoop对文本文件的通用称呼，CSV是一种使用逗号进行分隔的具体文本格式。
TextFile并不等于CSV，但是CSV是一种TextFile。

结构示例

```
1,Tom,20
2,Lucy,21
3,Jack,22
```

+ 优点
    + 格式非常简单
    + 可读性好：文本文件加之格式直观
    + 兼容性好：文本文件无需额外编解码或者解压缩
+ 缺点
    + 存储空间大：文本文件有编码效率低
    + 查询效率低：查询复杂度是O(n)，可能需要遍历整个文件
    + 不支持高效压缩（gzip）
        + 压缩后文件不支持拆分
        + 读取的解压操作，导致读取并行度下降
+ 适用场景
    + 小数据
    + 临时数据
    + 数据导入（其实也是临时数据的一种）

### SequenceFile

数据以二进制形式的行存储。
其中key一般是空的，而行的具体数据进行二进制处理保存在value里。

结构示例

```
<key,value>
```

+ 优点
    + 相比TextFile有更高的压缩率
        + Record Compression：每个value单独压缩
            + 压缩比低但是读写快
        + Block Compression：多行数据的key和value一起压缩
            + 压缩比高但是读写略慢
        + IO效率更高：因为压缩率更高
+ 缺点
    + 查询效率一般：因为key基本不起作用，查询复杂度依然是O(n)

### RCFile

RCFile是HIVE早期的列存储，但实际是行存储与列存储混合。
数据会先按行分组，每组内部按列存储。

+ 优点
    + 查询效率高：引入了列存储
+ 缺点
    + 结构复杂
    + 写入速度慢：列存储通病

### Parquet

Parquet是Apache开源的列存储格式，在不同系统中兼容性好，能用于Spark/Presto/Hive等。
Parquet沿用了RCFile的结构，数据会先按行分组，每组内部按列存储。

结构示例

```
File
 ├ RowGroup
 │   ├ ColumnChunk
 │   │   ├ Page
```

+ 若干行数据，组成一个RowGroup
    + 一个RowGroup大小通常为128MB
    + 每个RowGroup都有自己的RowGroupLevel，统计了min/max/sum/count等信息，用于谓词下推
+ 每个RowGroup内部，同一列数据被连续存储起来，组成一个ColumnChunk
+ 同一列的ColumnChunk，数据量可能很大，因此还会进行水平切分，分为多个Page
    + Page是最小的存储单元，也是最小的IO单元，每个Page都有独立的编码/压缩/索引

### ORC

ORC是HIVE开源的列存储格式，对HIVE有更好的优化。
ORC沿用了RCFile的结构，数据会先按行分组，每组内部按列存储。

结构示例

```
File
 ├ Stripe1
 │   ├ Index
 │   ├ Data
 |   |  ├ stream
 │   └ StripeFooter
 ├ Stripe2...
 ├ FileFooter
 └ PostScript
```

+ Stripe
    + 数据进行水平切割，默认256MB一块
    + 其中包含Index/Data/Footer三部分
    + Index
        + 包含该Stripe的min/max/sum/count等统计信息，用于谓词下推
    + Data
        + 实际存储数据的地方
        + 列存储，同一列的数据被连续存储
        + Data内部有多个stream，每个stream存储了该字段的某个属性
            + present stream记录该值是否是NULL
            + data stream存储真正的数据
            + length stream存储了该数据的长度（例如字符串长度）
    + StripeFooter
        + 记录该Stripe的结构，例如stream位置/长度/编码信息，用于在文件块中定位数据
        + stripe statistics：min/max/sum/count等统计信息，用于谓词下推
+ FileFooter
    + schema：表结构
    + column statistics：min/max/sum/count等统计信息，用于谓词下推
