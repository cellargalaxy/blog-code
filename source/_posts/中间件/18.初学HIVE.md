## 行存储与列存储

+ OLTP = On-Line Transaction Processing (联机事务处理)
    + 高频写入/整行查询/事务一致性
+ OLAP = On-Line Analytical Processing (联机分析处理)
    + 大量查询/聚合计算/查询部分列

### 行存储

+ 同一行的数据会被连续存储，在OLTP中有优势
+ 优点
    + 数据写入常见是按行写入，因此数据写入速度快，适合事务型操作
    + 由于同一行的数据会被连续存储，整行数据读取的效率高
+ 缺点
    + 如果只查询部分列，由于需要整行读取，数据读取效率较低
    + 同一行多种类型的数据连续存储，导致压缩率较低
+ 例如：TextFile/SequenceFile

### 列存储

+ 同一列的数据会被连续存储，在OLAP中有优势
+ 优点
    + 查询少量列，同一列的数据存储在一起，读取速度与IO效率效率高
    + 相同类型的数据被连续存储，压缩率高
    + 相同类型的数据连续存储，可以让CPU实现向量化处理
+ 缺点
    + 数据写入常见是按行写入，数据按列存储，需要对数据进行重排，导致写入较慢IO较多
    + 读取整行的效率较低
+ 例如：ORC/Parquet

## 存储格式

不同于数据库的存储引擎，存储引擎是一个独立的程序，而存储格式只是一种文件格式。

HIVE表的底层数据，作为一个文件，通常存储在HDFS/S3等分布式文件系统中，常见的文件格式如下

| 类型           | 存储方式 |
|--------------|------|
| TextFile     | 行存储  |
| SequenceFile | 行存储  |
| RCFile       | 行列混合 |
| Parquet      | 列存储  |
| ORC          | 列存储  |

### TextFile

TextFile是HIVE默认的存储格式，数据以文本形式的行存储。
格式类似于CSV，使用分隔符分隔列（分隔符可选），使用换行符分隔行。
TextFile是Hadoop对文本文件的通用称呼，CSV是一种使用逗号进行分隔的具体文本格式。
TextFile并不等于CSV，但是CSV是一种TextFile。

结构示例

```
1,Tom,20
2,Lucy,21
3,Jack,22
```

+ 优点
    + 格式非常简单
    + 可读性好：文本文件加之格式直观
    + 兼容性好：文本文件无需额外编解码或者解压缩
+ 缺点
    + 存储空间大：文本文件有编码效率低
    + 查询效率低：查询复杂度是O(n)，可能需要遍历整个文件
    + 不支持高效压缩（gzip）
        + 压缩后文件不支持拆分（split）
        + MapReduce读取时只能单线程解压，读取并行度下降
+ 适用场景
    + 小数据
    + 临时数据
    + 数据导入（其实也是临时数据的一种）

### SequenceFile

SequenceFile是Hadoop提供的一种二进制KV文件格式。
key为偏移量，一般是空的；value是一整行的二进制数据。

结构示例

```
<key,value>
```

+ 优点
    + 相比TextFile有更高的压缩率
        + Record Compression：每个value单独压缩
            + 压缩比低但是读写快
        + Block Compression：多行数据的key和value一起压缩
            + 压缩比高但是读写略慢
        + IO效率更高：因为压缩率更高
    + 支持文件拆分，便于MapReduce并行处理
+ 缺点
    + 查询效率一般：因为key基本不起作用，查询复杂度依然是O(n)
    + 仍然是行存储，无法跳过无关列

### RCFile

RCFile是HIVE早期的列存储，但实际是行存储与列存储混合。
数据会先按行分组，每组内部按列存储。

结构示例

```
File
├── RowGroup1
│   ├── Column1
│   ├── Column2
│   └── Column3
├── RowGroup2
│   ├── Column1
│   ├── Column2
│   └── Column3
└── ...
```

+ 优点
    + 查询效率高：引入了列存储，同一组的行都在一起，减少了数据重组的开销
    + 支持组内的列裁剪，减少IO
+ 缺点
    + 结构复杂，写入效率较低
    + 查询性能不如新一代格式，基本被Parquet/ORC取代

### Parquet

Parquet是Apache开源的列存储格式，在不同系统中兼容性好，能用于Spark/Presto/Hive等。
Parquet沿用了RCFile的结构，数据会先按行分组，每组内部按列存储。

结构示例

```
File
├── RowGroup1
│   ├── ColumnChunk1
│   │   ├── Page1 (DataPage/DictionaryPage)
│   │   └── Page2
│   └── ColumnChunk2
├── RowGroup2
└── FileMetadata
```

+ RowGroup：包含若干行数据
    + 一个RowGroup大小通常为128MB
+ ColumnChunk：由同一列数据组成
+ Page：同一列的ColumnChunk数据量可能很大，因此还会进行水平切分，分为多个Page
    + Page是最小的存储单元，也是最小的IO单元，每个Page都有独立的编码/压缩/索引
    + 常见的Page类型有两类
    + DictionaryPage：存储字典编码
    + DataPage：存储实际数据
+ FileMetadata
    + Schema结构
    + RowGroup列表
    + 每个ColumnChunk的统计信息，min/max/sum/count等，用于谓词下推

+ 优点
    + 使用列存储，支持谓词下推，查询部分列时IO更低性能更好
    + 列存储使用压缩率高
    + 生态支持更好：Spark/Presto/Flink/Trino
    + 支持Schema Evolution（新增列）
+ 缺点
    + 写入性能较低（相比行存）
    + 不支持数据更新，适合append场景
    + 索引能力弱于ORC，主要依赖FileMetadata中的统计信息，缺乏ORC多样化索引
+ 使用场景
    + Spark
    + Presto/Trino
    + 通用大数据分析平台

### ORC

ORC是HIVE开源的列存储格式，专为Hive优化设计。
ORC沿用了RCFile的结构，数据会先按行分组，每组内部按列存储。

结构示例

```
File
├── Stripe1
│   ├── Index
│   │   ├── RowGroup1
│   │   └── RowGroup2
│   ├── Data
│   │   ├── StreamA (present/data/length)
│   │   └── StreamB
│   └── StripeFooter
├── Stripe2
├── FileFooter
└── PostScript
```

+ Stripe
    + 数据进行水平切割，默认256MB一块
    + 其中包含Index/Data/Footer三部分
    + Index
        + 包含RowGroup级别的索引，min/max/sum/count等统计信息，用于谓词下推
        + RowGroup是一个逻辑概念，并不是一个客观结构，例如一万行组成一个RowGroup
        + 类似于Parquet的Page，将某一列水平切割为过个RowGroup
        + 包含该RowGroup在Data中的数据指针
    + Data
        + 实际存储数据的地方
        + 严格按列存储，RowGroup是逻辑概念，数据实际是被连续存储的
        + Data内部有多个stream，每个stream存储了该字段的某个属性
            + present：记录该值是否是NULL
            + data：存储真正的数据
            + length：存储了该数据的长度（例如字符串长度）
            + dictionary：存储字典本身
            + secondary：存储Decimal或Timestamp的辅助信息等
    + StripeFooter
        + 记录该Stripe的元信息，例如stream类型/位置/长度/编码信息，用于在文件块中解析定位数据
        + 无统计信息，不参与谓词下推
+ FileFooter
    + 记录了整个文件的元信息
    + schema：表结构
    + column statistics：记录了整个文件的统计信息，min/max/sum/count等，用于谓词下推
+ PostScript
    + 包含文件版本/压缩元数据/footer长度


+ 优点
    + 先进的索引机制：拥有文件级、Stripe级索引，结合布隆过滤器，谓词下推能力极强
    + 高压缩比：使用较为先进的编码与压缩类型
    + 支持ACID：支持事务性的插入更新删除
    + 内置统计信息：内置min/max等信息，无需扫描文件
+ 缺点
    + 生态相对局限：在Spark/Flink等非HIVE生态支持优先
    + 写入性能较低：需要构建负责的索引与统计数据

### Parquet与ORC比较

| 层级     | Parquet     | ORC      |  
|--------|-------------|----------|
| 文件水平切割 | RowGroup    | Stripe   |
| 列存储    | ColumnChunk | stream   |
| 列水平切割  | Page        | RowGroup |

| 生态支持   | ORC | Parquet |
|--------|-----|---------|
| Hive   | ✓✓✓ | ✓       |
| Spark  | ✓   | ✓✓✓     |
| Presto | ✓   | ✓✓      |
| Flink  | ✓   | ✓✓      |

+ HIVE推荐使用ORC
    + HIVE负责开发，优化最好
    + 压缩率高
+ Spark推荐使用Parquet
    + 是Spark的默认格式
    + 执行向量化读取
