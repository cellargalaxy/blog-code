## 简介

+ HDFS（Hadoop Distributed File System），是Hadoop的分布式文件系统
+ 起源于2003年谷歌发布的论文The Google File System（GFS），是GFS的简化开源版本
+ HDFS的设计目的
    + 运行在普通廉价的服务器上
    + 存储超大规模的数据
    + 追求高吞吐而不是低延迟
+ HDFS经过了四个大版本更新，目前常用的是2.x与3.x
    + 0.x：最古早的实验版本
    + 1.x：单NameNode架构，存在单点故障
    + 2.x：支持NameNode高可用（一Active/一Standby）；引入yarn
        + yarn是一个资源调度系统，与HDFS同属Hadoop生态的不同组件
    + 3.x：升级NameNode高可用（一Active/多Standby）；支持擦除编码；最低JAVA8
+ 使用场景：一次写入，多次读取
    + 文件写入完成关闭后，通常只读
    + 只支持追加操作（append）
+ 向HDFS存储文件，客户端按照Block大小流式发送数据，DataNode按照Block大小进行存储，以支持存储超大文件
    + Block的大小默认128M，较大的Block有助于降低寻址开销，也意味着HDFS适合于存储大文件而不是小文件
    + 如果文件大于128M，客户端负责将文件切割为多个不大于128M的Block
    + 如果文件不足128M，则只生成一个Block，一个10M的文件生成的Block也只是10M
+ HDFS的优点
    + 通透性：尽管文件实际是分块存储，但客户端使用就像访问本地磁盘一样
    + 高容错：HDFS的数据会保存多个副本（默认3份），即便部分机器宕机，系统整体也依然高可用且不会导致数据丢失
    + 高吞吐：适合大数据量的顺序读写
    + 性价比：能运行在大量廉价机器上
+ HDFS的缺点
    + 不适合低延迟数据访问：数据读写延迟比较高，无法实现毫秒级读写
    + 不适合大量小文件存储：大量小文件意味着大量小Block，会增大NameNode的存储压力
    + 不支持并发写入与随机写入：一个文件同一时间只支持一个线程写入，只支持对文件执行追加操作

## 架构

以Hadoop3.x为例，HDFS架构为主从结构，包括主节点与从节点

```
NameNode  -> 元数据管理
DataNode  -> 数据存储
Client    -> 访问入口
```

```
           +----------------+
           |    Client      |
           +--------+-------+
                    |
               (metadata)
                    |
           +-------v-------+
           |   NameNode    |
           +-------+-------+
                   |
       ----------------------------
       |           |            |
  +----v----+ +----v----+ +----v----+
  |DataNode | |DataNode | |DataNode |
  +---------+ +---------+ +---------+
```

+ 通讯协议
    + 各个组件之间基于TCP/IP协议，使用RPC协议进行远程调用
+ NameNode（名称节点）
    + 主节点
    + 负责维护文件系统的元数据，以及响应客户端对文件的访问
    + 元数据包括目录树/文件权限/文件的Block列表/Block与DataNode的映射关系/副本数据
    + 元数据包含两份数据
        + FsImage：元数据的只读快照，会进行持久化，是文件系统在某一时刻的完整快照，但是数据是落后的
        + Edits：元数据的编辑日志，会进行持久化，NameNode收到元数据的编辑请求，会将编辑情况记录到Edits中
        + NameNode冷启动时，会加载FsImage与Edits到内存中，然后重放Edits，以得到最新的元数据
        + 在NameNode运行时，所有元数据都在内存中，因此内存大小会限制集群规模
    + 安全模式
        + NameNode刚刚启动时，会自动进入只读状态
        + 在提供服务前，需要等待DataNode上报Block信息，确认Block的副本数是否达标
        + 当99.9%的Block都满足最低副本要求（默认是1），30秒后会自动退出安全模式，允许正常读写
+ DataNode（数据节点）
    + 从节点
    + 负责存储实际的数据，处理客户端对Block数据的读写
    + 会周期性向NameNode发送心跳
        + 心跳（Heartbeat）：默认每3秒，用于表示节点存活，连续10min都心跳丢失，DataNode会被认为宕机，不再被分配到IO操作
        + 块报告（Block Report）：默认每小时，用于报告所存储的全部Block，让NameNode获取全部Block信息，如果有Block损坏及时创建新副本
+ SecondaryNameNode（第二名称节点）
    + 辅助节点
    + 不是NameNode的备用节点
    + 负责将Edits合并到FsImage，并将新FsImage更新到NameNode中，避免Edits无限增长，拖累冷启动时间
        + 每隔一段时间，SecondaryNameNode会要求NameNode切换到新Edits文件
        + SecondaryNameNode将NameNode的FsImage与旧Edits拉回本地内存，进行合并操作
        + 将合并后的FsImage同步回给NameNode
        + NameNode使用新替换旧FsImage，并且删除旧Edits
+ JournalNode（日志节点）
    + 辅助节点
    + 在NameNode的高可用模式时使用，通常部署奇数个（3个/5个）
    + 用于维护NameNode实例之间的元数据的一致性
    + ActiveNameNode写入Edits时，必须写入多数JournalNode
+ ZKFC（Zookeeper Failover Controller）
    + 辅助节点
    + 在NameNode的高可用模式时使用
    + 用于监控ActiveNameNode的存活，进行自动故障切换

## 读写流程

### 写入流程

+ Client请求NameNode创建文件，NameNode检查文件是否存在，检查是否有权限，然后在元数据中创建该文件的记录
+ Client告知NameNode，每个Block需要三个DataNode来存储3个副本
+ NameNode通过机架感知机制，获取三个DataNode
    + DataNodeA：如果客户端在集群中，则客户端作为第一个DataNode，否则随机取一个DataNode
    + DataNodeB：在DataNodeA以外的机架中，随机取一个DataNode
    + DataNodeC：在DataNodeB机架中，随机取一个DataNode
    + 同一机架共享电源/网线/交换机，不同副本存储在不同机架上避免整个机架宕机
+ NameNode将三个DataNode告知客户端
+ 客户端按照Block大小流式发送数据，客户端将Block发送给DataNodeA进行存储
    + DataNodeA将Block同步给DataNodeB
    + DataNodeB将Block同步给DataNodeC
    + 通过机房内部数据同步，减少客户端数据传输
    + 传输完成后反向逐级进行ACK确认，以确保全部副本的写入成功
    + DataNodeC->DataNodeB->DataNodeA->Client
    + 如果其中有DataNode保存失败，NameNode会检查到副本数量不足，触发副本复制操作
+ 直到全部Block都存储完成，客户端告知NameNode完成写入

### 读取流程

+ 客户端将文件名称发送给NameNode
+ NameNode根据文件名称查询到该文件的全部Block，以及这些Block存储在哪些DataNode中
+ 将Block与DataNode信息返回给客户端
+ 客户端直接访问最邻近的DataNode，读取文件
+ 每个Block都会使用校验和来校验数据完整性，如果发现数据损坏，会向NameNode报告，然后换另外一个DataNode重新读取

## 高可用

```
Client
   ↓
Active NN     Standby NN
     \        /
     ↓        ↑
      \      /
    JournalNodes
```

+ 3.x支持在一个集群上，部署多个NameNode实例
+ 任意时间点只有一个NameNode是Active状态，其余均为Standby状态
+ Active负责处理全部请求，Standby会同步Active的元数据，并且进行合并
    + Standby提供快速恢复能力
    + 不能再使用SecondaryNameNode
+ Active与Standby的元数据一致性
    + 元数据有两类，静态的是FsImage与Edits，动态的是Block与DataNode关系
    + FsImage只读，因此只需要保证Edits的事务性同步，通过JournalNode实现
    + DataNode会向全部NameNode实例，上报自己的Block，因此每个NameNode的动态数据是时刻同步的
+ 当Active收到元数据的编辑请求时
    + Active将操作日志持久化到多数JournalNode的Edits里
    + Active更新自己内存中的Edits
    + 返回客户端成功
    + Active异步持久化自己的Edits
    + Standby通过JournalNode，实时获取到最新的Edits，保持与Active的同步
    + 一旦Active宕机，Standby需要保证从JournalNode中拉取到最新的Edits，然后进行选主
+ 选主
    + NameNode启动的时候会向zookeeper注册一个临时节点，如果Active挂了，Active的临时节点也会消失
    + zookeeper会有个监视器监视到，会将原NameNode置为Standby，选择另外一个NameNode置为Active

## 高扩展

+ 通过联邦HDFS（Federation）架构，每个NameNode实例只负责一部分目录的元数据
+ 能通过拆分目录大小，控制单个NameNode所承载的元数据大小，避免受限于单台NameNode的内存大小
+ 多台NameNode能同时提供服务，提高吞吐
+ 不同业务数据交由不同的NameNode与DataNode管理，减少业务间的影响
